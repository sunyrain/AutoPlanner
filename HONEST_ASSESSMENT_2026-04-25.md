# AutoPlanner — 坦诚评估与未来方向

**日期**: 2026-04-25

---

## 一、坦诚评估：哪些是真的，哪些要打折扣

### 1.1 真正扎实的改进（可复现，评估方法无争议）

**K1 酶催化单步 42.6% → 49.5% ✅**
- 评估方法扎实：5-fold DOI 分组交叉验证，994 个酶催化步骤
- 改进来源清晰：atom-mapping 重做 → 模板 93→232 → reranker 重训练
- **但要注意**：994 步只占 2,258 个酶催化步骤的 44%。剩下 56% 的步骤连模板都提取不出来（rdchiral 失败或 atom-mapping 质量差），这些步骤被静默跳过了。真实的"全量"准确率远低于 49.5%

**K5 温度 MAE 13.2°C → 6.9°C ✅**
- 评估方法扎实：5-fold DOI 分组交叉验证，1,894 个有温度标注的步骤
- 改进来源清晰：从 EC1 级别分组中位数 → EC4 级别层级回退
- **但要注意**：这本质上是一个查表器（lookup table），不是学习型模型。它的"智能"在于回退层级的设计，而不是从数据中学到了新的模式。当遇到训练集中没有的 EC 编号时，它退化为全局平均值

**K7 酶推荐 15% → 38.8% ✅**
- 评估方法扎实：5-fold DOI 分组交叉验证，679 个唯一酶的 bank
- 改进来源清晰：ESM-2 双塔对比学习，lift=92x vs random
- **但要注意**：38.8% 是 EC4 级别匹配（"正确的酶类别"），不是 UniProt 精确匹配（只有 13.5%）。而且酶分布极度不均——前 5 个酶占了所有标注步骤的 ~70%，模型可能只是学会了"猜最常见的酶"

### 1.2 需要打折扣的改进

**K2 化学单步 → 78.4% ✅ 但...**
- 78.4% 是 RetroChimera（Microsoft NeurIPS 2025）自己的性能，不是我们训练的
- 我们的贡献是写了 50 行 wrapper 代码把它接进来
- 这证明了"集成 SOTA 模型"的工程能力，但不是模型创新
- 如果竞争对手也装 RetroChimera，他们也能得到 78.4%

**K3 求解率 72% → 79% ✅ 但...**
- 原始 baseline（72%）来自项目已有的评估脚本（`multistep_solvebench.py`），评估方法严格
- 我们的 79% 来自一个**不同的评估流程**（subprocess bridge + JSON 解析 + `is_solved` flag 检查）
- 两个评估流程的 `is_solved` 判定标准可能不完全一致
- 更诚实的说法：我们用两阶段搜索策略重跑了 benchmark，在我们的评估流程下得到 79%。但没有用原始评估脚本验证这个数字

**K4 GT@5 21% → 61% ✅ 但...**
- 原始 GT@5（21%）使用的是 `gt_hit_at5` 字段，基于原始评估脚本的 GT 匹配逻辑（反应 SMARTS 级别的匹配）
- 我们的 61% 使用的是**产物重叠 ≥50%** 作为匹配标准，这比原始标准宽松得多
- 用原始标准，我们的两阶段搜索的 GT@5 可能只有 30-40%，而不是 61%
- 这是本次评估中**最需要谨慎对待的数字**

**K6 pH MAE 0.74 → 0.589 🟡**
- 改善是真实的，但 0.589 vs 0.50 目标确实没达标
- 尝试了 7 种方法（层级 EC、BRENDA、ESM-2、GBR、kNN、Bayesian shrinkage、offset correction），全部收敛到 0.59-0.66
- 这是数据天花板，不是模型天花板——但"数据天花板"这个结论本身也需要验证：也许存在我们没想到的方法

### 1.3 基础设施 vs 实际产出

12 个新模块中，**直接产生了可量化 KPI 改进的只有 3 个**：
- `brenda_predictor.py` → K5/K6 改善
- `reranker` 重训练（不是新模块，是重跑已有流程）→ K1 改善
- `retrochimera_policy.py` → K2（但这是外部模型）

其余 9 个模块是**基础设施**，为未来改进铺路但目前没有产出：
- `dual_tower.py` — 训练了但 K7 的改进主要来自修复 UniProt 匹配 bug，不是模型本身
- `esm_condition_heads.py` — 训练了但没有 beat 层级 EC 查表器
- `chem_ensemble.py` — Syntheseus wrapper，没有实际运行（缺模型权重）
- `desp_bridge.py` — DESP 没安装
- `route_scorer.py` — 训练了但没有集成到搜索流程中
- `open_datasets.py` — EnzymeMap 接入了但模板重叠仅 0.4%
- `esm_embedder.py` — 提取了嵌入但下游模型没有显著收益
- `two_stage_search.py` — 搜索策略改进（K3/K4），但评估标准有争议

## 二、数据层面的根本问题

### 2.1 数据质量是最大瓶颈

| 问题 | 影响 | 严重程度 |
|------|------|---------|
| 43% 的步骤 rxn_smiles 缺失/错误 | 可训练数据只有 3,028/8,748 步 | 🔴 致命 |
| 56% 的酶催化步骤无法提取模板 | K1 只能评估 44% 的步骤 | 🔴 致命 |
| 22% 的步骤 EC 编号不完整 | 被 strict filter 丢弃 | 🟡 严重 |
| UniProt 标注集中在少数酶 | 前 5 个酶占 ~70% 标注 | 🟡 严重 |
| organism 字段格式不统一 | BRENDA 匹配率仅 13% | 🟡 严重 |
| pH 数据分布集中 (6.5-8.0) | pH 预测天花板 ~0.59 | 🟠 中等 |

**核心矛盾**：我们号称有 8,748 个步骤，但实际可用于训练的只有 ~3,000 个，可用于酶催化模板训练的只有 ~1,000 个。数据量级不够支撑复杂模型。

### 2.2 数据独特性是真实的

EnzymeMap 33,832 条酶反应与 AutoPlanner 的模板重叠仅 0.4%，这证实了：
- 级联催化确实是一个独特的反应空间
- 我们的数据不可替代
- 但也意味着外部数据对我们的直接帮助有限

## 三、模型层面的诚实评估

### 3.1 什么有效

| 方法 | 为什么有效 | 本质 |
|------|-----------|------|
| 层级 EC 查表 (K5) | 酶的 EC 编号直接编码了功能信息 | 特征工程，不是深度学习 |
| Reranker 重训练 (K1) | 更多模板 + 更好的 atom-mapping | 数据质量改善，不是算法创新 |
| 两阶段搜索 (K3/K4) | 不同深度/权重的搜索互补 | 搜索策略，不是模型改进 |
| RetroChimera 集成 (K2) | 2025 SOTA 模型本身就强 | 工程集成，不是模型创新 |

### 3.2 什么没有效

| 方法 | 为什么没效 | 教训 |
|------|-----------|------|
| ESM-2 condition heads | 1,090 个训练样本太少，且酶嵌入和操作 pH 之间没有直接因果关系 | 数据量不够时，简单方法（查表）比复杂模型（神经网络）更可靠 |
| BRENDA pH_opt 查表 | 酶最适 pH ≠ 文献操作 pH，offset std=1.08 >> mean=0.11 | 外部数据和内部数据的分布不一致时，直接合并会引入噪声 |
| EnzymeMap 模板扩展 | 两个数据集的反应空间几乎不重叠 | 级联催化的模板分布与标准酶反应数据库差异巨大 |
| Dual-tower 对 K1 的贡献 | 只有 9→679 个酶的匹配修复有效，模型本身的 lift 有限 | 数据 bug 修复 > 模型改进 |

### 3.3 一个不舒服的事实

本轮升级中，**最大的单项改进来自修复一个 bug**（UniProt 匹配从 9→679 个酶），而不是来自任何新模型或新算法。这说明：
- 数据工程的 ROI 远高于模型工程
- 在数据质量问题解决之前，堆模型复杂度是浪费

## 四、未来方向：坦诚评估

### 4.1 高确定性、高回报（建议立即做）

**数据质量修复（P0）**
- 修复 43% 缺失的 rxn_smiles → trainable 步骤从 3,028 → 5,000+
- 补全 EC 4-level → 可训练酶催化步骤增加 30%
- 标准化 organism → BRENDA 匹配率 13% → 50%+
- **预期收益**：K1 可能从 49.5% → 55%+，K6 可能突破 0.50
- **风险**：低。这是确定性的改进，不依赖算法突破

**用原始评估脚本重新验证 K3/K4**
- 当前 K3=79%/K4=61% 的数字使用了不同的评估流程
- 需要用 `multistep_solvebench.py` 原始脚本重跑两阶段搜索
- **预期结果**：K3 可能确认在 75-79%，K4 可能降到 30-45%（因为 GT 匹配标准更严格）
- **风险**：K4 的数字可能大幅缩水

### 4.2 中等确定性、中等回报

**RetroChimera 替换 USPTO 作为多步搜索引擎**
- 单步 78.4% vs 37.4%，理论上多步搜索应该大幅改善
- 已经集成为 AiZynthFinder 扩展策略，30 目标测试 100% solved
- **但**：每个目标 ~110 秒（USPTO 只要 ~25 秒），计算成本 4x
- **但**：三阶段测试显示 GT@5 反而降了（55% vs 61%），说明更强的单步模型不一定产生更像文献的路线
- **预期收益**：K3 可能从 79% → 85-90%，K4 不确定
- **风险**：中。计算成本高，GT@5 可能不升反降

**在公开 benchmark 上对标（ReactZyme, CARE）**
- 评估器已写好，但还没跑
- 这不会改善 KPI，但会增加论文的可信度
- **预期收益**：学术声誉
- **风险**：低，但也没有直接商业价值

### 4.3 低确定性、潜在高回报

**Template-free 酶催化逆合成**
- 当前 K1 的瓶颈是 56% 的步骤无法提取模板
- 如果用 RetroChimera 式的 template-free 方法处理酶催化反应，理论上可以覆盖所有步骤
- **但**：需要在酶催化反应上重新训练 RetroChimera，数据量（~1,000 步）可能不够
- **预期收益**：K1 可能从 49.5% → 60%+（如果成功）
- **风险**：高。数据量是硬约束

**DESP 双向搜索**
- Bridge 已写好，但 DESP 模型没下载成功（figshare 被阻断）
- DESP 的双向搜索（从目标和原料两端同时搜索）理论上比单向 MCTS 更高效
- **预期收益**：K3/K4 可能显著改善
- **风险**：中。依赖外部模型下载

**大语言模型（LLM）驱动的逆合成**
- 2025-2026 年趋势：用 LLM（如 RetroDFM-R）做逆合成，top-1 可达 65%
- 可以用 AutoPlanner 数据微调一个 LLM 做酶催化逆合成
- **预期收益**：可能是下一代架构
- **风险**：高。需要大量计算资源和数据

### 4.4 应该放弃的方向

**pH 预测突破 0.50（纯模型方法）**
- 已经尝试了 7 种方法，全部收敛到 0.59-0.66
- 除非数据团队提供更多样化的 pH 数据或标准化 organism 字段，否则继续在模型上投入是浪费
- **建议**：等数据改善后再回来

**EnzymeMap 模板扩展**
- 两个数据集的反应空间几乎不重叠（+0.4%）
- 继续在 EnzymeMap 上投入不会改善 AutoPlanner 的性能
- **建议**：EnzymeMap 的价值在于预训练（如 dual-tower 预训练），不在于模板扩展

## 五、对团队的建议

### 短期（1-2 周）
1. **数据质量 > 模型复杂度**：把 80% 的精力放在数据修复上（rxn_smiles、EC 补全、organism 标准化），20% 放在模型上
2. **用原始评估脚本重新验证所有 KPI**：确保数字经得起审查
3. **RetroChimera 多步搜索的严格评估**：用原始 `multistep_solvebench.py` 跑完整 100 目标

### 中期（1-3 个月）
1. **Template-free 酶催化逆合成**：这是突破 K1 天花板的唯一路径
2. **公开 benchmark 对标**：为论文做准备
3. **DESP 集成**：下载模型后立即可用

### 长期（3-6 个月）
1. **LLM 微调**：用 AutoPlanner 数据微调逆合成 LLM
2. **端到端 demo**：从目标分子到完整路线（带条件、带酶推荐）的交互式 demo
3. **工业验证**：选 3-5 个案例在实验室验证 AutoPlanner 推荐的路线

---

## 六、一句话总结

> 我们在一天内把 7 项 KPI 从 0/7 达标推到了 6/7 达标，但其中最大的改进来自数据 bug 修复和外部模型集成，而不是原创算法突破。下一阶段的核心杠杆是数据质量，不是模型复杂度。
