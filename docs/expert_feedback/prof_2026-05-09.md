你真正的创新点应该重新表述为：

> **AutoPlanner 不是普通 chemoenzymatic retrosynthesis planner，而是一个 condition-constrained catalytic cascade planner：它要同时规划多步催化路线、判断哪些步骤能被合并为级联过程、分配共享或分段条件，并显式评估 pH / 温度 / 溶剂 / buffer / cofactor / catalyst incompatibility。**

这比“接近国际前沿”更有冲击力。因为现在已有前沿系统在做 hybrid organic–enzymatic planning、酶推荐、条件预测、plausibility evaluation 和 active-site validation；例如 ChemEnzyRetroPlanner 已经把 hybrid retrosynthesis、condition prediction、enzyme recommendation 和 active-site validation 放到一个平台里。([Nature][1]) ACERetro 也已经用 synthetic potential score 和 asynchronous search 做 chemoenzymatic planning，并在 1001 molecules 测试集上报告能找到比已有 SOTA 多 46% 的 hybrid routes。([化学文献共享平台][2]) TTLAB 则把有机和酶促 transformer loop 结合，用 best-first tree search 做 chemoenzymatic multistep retrosynthesis。([化学文献共享平台][3])

所以，如果你继续把 AutoPlanner 描述成“多源候选 + route-tree + verifier”，它会显得像已有 CASP 系统的组合；但如果你把核心任务定义成 **级联催化过程规划**，重点就变了。

---

## 先说结论：数据缺失不是阻碍，而是你的 SOTA 机会

级联催化数据缺失非常严重，这是真的。但这恰恰说明：**不能靠普通 supervised learning 路线做 SOTA**。如果你试图收集一个大规模“target → cascade route + condition”的标准训练集，再训练一个大 transformer，很可能失败，因为真实 cascade 样本太少、报道格式不统一、负样本几乎没有、条件记录也不完整。

更好的方向是：

> **用少量 gold cascade 数据 + 大量 single-step reaction / enzyme / condition 数据 + weak supervision + 物理化学约束 + active learning，构建一个 data-scarce cascade planner。**

你的系统要变成“在数据稀缺下仍能规划级联催化”的模型，而不是“等数据足够后再训练的模型”。

这才有机会成为最先进模型。

---

## 你现在不应该追求一个“大一统模型”

你现在最危险的想法是：
“既然数据少，我是不是应该想办法造数据，然后训练一个端到端 cascade transformer？”

不建议这样做。

级联催化不是简单的序列预测。它至少包含五个相互耦合的问题：

1. **多步反应路径是否成立。**
2. **每一步的催化剂或酶是否可能工作。**
3. **这些步骤能否在同一条件下共存。**
4. **是否需要 one-pot、sequential one-pot、telescoping、buffer exchange、immobilization 或分段操作。**
5. **cofactor、pH、温度、溶剂、金属、氧化还原体系、抑制剂、副产物是否互相冲突。**

所以你真正需要的不是一个普通路线模型，而是一个 **cascade-state model**。

中文可以叫：

这比“chemoenzymatic route planner”更尖锐，也更容易形成独立贡献。

你的论文或项目主张可以是：

> 现有 chemoenzymatic planning 系统主要解决“有机反应和酶反应如何混合成路线”，但很少把级联催化的共享条件、步骤兼容性、cofactor ledger、stage partition 和实验过程约束作为搜索状态的一部分。AutoPlanner-Cascade 将 cascade condition state 显式纳入 route-tree search，并在数据稀缺下通过 weak supervision、知识图谱、规则约束和专家/实验反馈训练可校准的 cascade feasibility model。

## 最核心的建模对象：condition envelope，不是 single-step condition

普通 condition prediction 是：

```text
reaction → condition
```

你的问题应该是：

```text
route tree + catalysts + enzymes + stage partition → feasible condition envelope
```

例如一条路线有 4 步：

```text
Step 1: transaminase
Step 2: alcohol dehydrogenase
Step 3: lipase
Step 4: Pd-catalyzed coupling
```

普通系统会分别预测每一步条件。
你的系统要回答：

```text
Step 1-2 是否能 one-pot？
Step 3 是否可以接在同一 buffer 里？
Pd step 是否必须隔离？
是否需要 NADH regeneration？
胺供体是否抑制 ADH？
有机溶剂比例是否会让酶失活？
pH 7.5 是否同时满足 transaminase 和 ADH？
温度 30 °C 是否牺牲 lipase 活性？
如果分成两个 stages，哪个切点最合理？
```

这才叫真正处理级联催化。

## 数据缺失时，应该怎么做

### 1. 不要等待“大数据集”，先建立 CascadeDB-Gold

你需要自己定义一个小而高质量的 gold set。

不要一开始追求 10 万条。
先追求 200–500 条高质量级联记录。

每条记录至少标注：

```text
target / product
cascade type: enzymatic / chemoenzymatic / organocatalytic / metal-enzyme / photo-enzyme
number of steps
one-pot or sequential
step reactions
intermediates
catalyst / enzyme / EC / organism
pH
temperature
solvent / buffer
cofactor
cosubstrate
additives
yield / conversion / ee / dr
stage partition
whether isolation occurred
reported incompatibilities
literature source
```

这里的 gold set 不是为了训练大模型，而是为了：

1. 定义任务。
2. 校准模型。
3. 做 benchmark。
4. 给实验组审查。
5. 证明你解决的是一个没人系统解决好的问题。

SOTA 不一定来自最大数据集，很多时候来自**最清晰的新任务定义 + 最可信 benchmark**。

---

### 2. 用 public biochemical databases 构造 silver cascade 数据

你可以从这些来源构造 weak supervision：

| 数据源                          | 用法                                                                                                                                                                  |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Rhea                         | 获取 expert-curated biochemical reactions、EC、ChEBI 标准化反应参与物；Rhea 是 UniProtKB 酶和 transporter annotation 的标准反应知识库。([Rhea][4])                                           |
| BRENDA                       | 获取酶信息、pH、温度、底物、cofactor、酶性质；BRENDA 2026.1 release 已更新到 2026 年 3 月，并有 knowledge graph / SPARQL endpoint。([BRENDA酶数据库][5])                                            |
| SABIO-RK                     | 获取 kinetic parameters 和 experimental conditions；SABIO-RK 明确包含 biochemical reactions、kinetic equations、parameters 和 experimental conditions。([sabiork.h-its.org][6]) |
| MetaCyc                      | 获取 experimentally elucidated metabolic pathways、enzymes、metabolites、reactions，用于构造自然 pathway adjacency。([metacyc.org][7])                                           |
| ORD / USPTO                  | 获取有机反应条件、催化剂、溶剂、温度等，用于有机步骤 condition prior；ORD 的目标就是支持 reaction prediction、synthesis planning 和 experiment design。([docs.open-reaction-database.org][8])            |
| ECREACT / biocatalysis-model | 用于酶促反应 prediction / retrosynthesis 的预训练或 proposal source；RXN biocatalysis-model 使用 ECREACT 做生化反应预测。([GitHub][9])                                                    |

但是要注意：这些 silver 数据不是直接等价于“实验级 cascade”。TTLAB 论文也明确指出，许多 biochemical pathway 数据只部分反映有机合成中的酶应用，因为有机合成里的酶可能在非天然条件、高底物浓度、固定化形式、非天然底物下使用。([化学文献共享平台][3])

所以你要把它们标成 **weak / silver evidence**，不能当 gold truth。

---

### 3. 用规则和反事实样本构造负样本

级联建模最缺的是负样本：文献通常报道成功级联，很少系统记录“这个组合不兼容”。

你可以构造负样本，但必须透明标注为 synthetic negative / rule-derived negative。

例如：

| 负样本类型        | 构造方式                                                   |
| ------------ | ------------------------------------------------------ |
| pH 不兼容       | enzyme A optimum pH 5–6，enzyme B optimum pH 9–10，无共同窗口 |
| 温度不兼容        | 一个酶需要 20–30 °C，另一个催化步骤需要 >100 °C                       |
| 溶剂冲突         | 高比例 DMF / DMSO / MeCN 与不耐有机溶剂酶组合                       |
| 金属冲突         | Pd/Cu/Ni 与敏感酶或 thiol-containing system 冲突              |
| 氧化还原冲突       | 强氧化剂与还原酶 / NADH 系统共存                                   |
| cofactor 不闭合 | NADH/NADPH 消耗无 regeneration                            |
| 中间体毒性        | aldehyde、quinone、peroxide 等可能抑制酶                       |
| buffer 冲突    | phosphate、Tris、amine buffer 与金属催化或活化试剂冲突               |

这些负样本的价值不是“证明一定失败”，而是训练模型知道：**缺证据时不要盲目把所有步骤合并成 one-pot cascade。**

---

### 4. 建立 compatibility graph，而不是只做 compatibility score

你现在的 compatibility_success_any 还像一个路线级指标。未来要变成图结构：

```text
nodes = steps / catalysts / enzymes / cofactors / solvents / buffers / intermediates
edges = compatible / risky / incompatible / unknown
```

例如：

```text
enzyme_1 --compatible_with_pH--> pH 7.0-8.0
enzyme_2 --compatible_with_pH--> pH 7.5-9.0
Pd catalyst --inhibited_by--> thiol
NADH system --incompatible_with--> strong oxidant
intermediate aldehyde --risk_of--> enzyme inhibition
```

然后 cascade planner 不只是问：

```text
这条路线分数高不高？
```

而是问：

```text
这个 compatibility graph 是否存在一个可行 condition assignment？
```

这可以变成一个约束满足 / probabilistic factor graph 问题。

## 模型层面：你应该训练什么

我建议把模型拆成 6 个头，而不是一个 end-to-end 输出路线的模型。

### 1. Step Plausibility Head

判断某个单步催化反应是否合理：

```text
P(step works | product, reactants, catalyst/enzyme, EC, evidence)
```

它可以利用现有 organic / enzymatic reaction data 训练。

---

### 2. Catalyst / Enzyme Match Head

判断某个 enzyme 或 catalyst 是否适合该底物：

```text
P(catalyst matches substrate transformation)
```

输入可以包括：

```text
reaction fingerprint
substrate/product fingerprint
EC hierarchy
enzyme sequence embedding
active-site features
substrate similarity
organism/source
literature evidence
```

如果没有具体 enzyme sequence，就退化到 EC-level confidence。

---

### 3. Condition Envelope Head

输出不是一个点条件，而是一个窗口或分布：

```text
pH: 6.8–8.2
T: 25–37 °C
solvent: aqueous buffer + <=10% DMSO
buffer: phosphate / Tris risk noted
cofactor: NADP+, glucose dehydrogenase regeneration recommended
```

反应条件预测本身就是前沿难题。2025 年的 Chemical Science perspective 明确指出，condition prediction 在 CASP 中仍面临 data quality、sparsity、reaction representation 和 evaluation challenges，甚至有研究显示模型可能不超过 literature-derived popularity baselines。([化学文献共享平台][10]) 单步条件预测已经有进展，例如 Reacon 在 refined USPTO 上报告 recorded conditions top-3 accuracy 63.48%，cluster level top-3 accuracy 85.65%。([化学文献共享平台][11]) 但你的目标不是单步 top-k condition，而是多步共享 condition envelope。

---

### 4. Pairwise Compatibility Head

判断两个步骤能否放在同一 stage：

```text
P(step_i compatible_with step_j | condition envelope)
```

这个头非常关键，因为 cascade 的本质不是“多步”，而是“多步共存”。

训练数据可以来自：

```text
gold cascade 中共存的 step pairs = positive
自然 pathway adjacency = weak positive
文献 one-pot / tandem / cascade = positive
规则冲突组合 = synthetic negative
专家标注 = high-value labels
```

---

### 5. Global Cascade Value Head

判断整个 cascade plan 的成功概率：

```text
P(cascade feasible | route, stages, conditions, catalysts, evidence)
```

这不是普通 route value。
普通 route value 关注“能否从 stock 合成 target”。
cascade value 关注：

```text
能否在给定 stage partition 和 condition envelope 下实际运行？
```

可以写成近似分解：

```text
log P(cascade feasible)
≈ Σ log P(step works)
+ Σ log P(catalyst active under condition)
+ Σ log P(pairwise compatible)
+ log P(cofactor/redox closed)
+ log P(stage operation feasible)
- uncertainty penalty
```

---

### 6. Experiment Selection / Active Learning Head

这是你冲 SOTA 的关键。

数据少时，模型要主动告诉实验组：

```text
最值得做哪一个小实验？
哪个 step pair 的兼容性最不确定？
哪个 pH / solvent / cofactor 条件最能区分路线可行性？
哪 5 个 enzyme candidates 最值得筛？
```

这会让系统从“路线建议工具”变成“实验协同优化系统”。

---

## 训练策略：三阶段，而不是直接端到端

### 阶段 1：预训练单步知识

用大规模 single-step 数据训练：

```text
reaction representation
enzyme / EC representation
condition prior
product-reactant compatibility
stock / route features
```

这里可以用 Rhea、BRENDA、SABIO-RK、MetaCyc、ECREACT、ORD、USPTO 等来源。不要指望这些数据直接教会 cascade，但它们可以教会模型“单步反应和条件的基础语言”。

### 阶段 2：weakly supervised cascade compatibility

构造 silver cascade graph：

```text
natural pathway adjacency
published multi-enzyme cascades
chemoenzymatic one-pot examples
synthetic negative incompatibility pairs
expert-labeled risky pairs
```

训练：

```text
pairwise compatibility
condition overlap
stage merge/split decision
cofactor closure
global cascade value
```

这一步是你和普通 CASP 拉开差距的地方。

---

### 阶段 3：human-in-the-loop / wet-lab active learning

每一轮让模型选：

```text
10 个最有信息量的 step pairs
10 个最有不确定性的 condition windows
5 条高潜力但低证据 cascade plans
```

实验组做小规模验证后回流：

```text
works / fails
conversion
yield
ee / dr
enzyme activity retained?
inhibition observed?
condition notes
```

然后更新 compatibility model 和 uncertainty model。

这比盲目扩大文献数据更有效。

## 你需要建立自己的 benchmark：CascadeBench

如果你想做“最先进”，不要只在 USPTO、PaRoutes、Retro*-190 这类普通 retrosynthesis benchmark 上竞争。那些 benchmark 不能证明你会处理级联催化。

你应该建立一个新 benchmark：

> **CascadeBench: condition-constrained catalytic cascade planning benchmark**

它可以分成四个任务。

### Task 1：Cascade Route Recovery

输入 target，输出多步催化 route。

指标：

```text
step recovery@k
key disconnection recovery@k
enzyme/EC recovery@k
route semantic similarity
```

---

### Task 2：Stage Partition Prediction

输入已知 route，判断哪些步骤可以合并：

```text
one-pot
sequential one-pot
telescoped
requires isolation
requires buffer exchange
```

指标：

```text
stage partition F1
merge/split accuracy
one-pot feasibility accuracy
```

---

### Task 3：Condition Envelope Prediction

输出多步共享条件窗口：

```text
pH interval IoU
temperature interval IoU
solvent/buffer class accuracy
cofactor requirement accuracy
condition top-k recall
```

这里不要只用 exact match。条件预测用 exact string match 很不合理。

---

### Task 4：Cascade Feasibility Ranking

给定多个 candidate routes，排序哪个最可能实验可行。

指标：

```text
expert preference accuracy
reported successful cascade rank
calibrated feasibility ECE
Brier score
wet-lab hit rate@k
```

这才是真正能体现你系统先进性的 benchmark。

## 当前最应该做的 5 件事

### 第一，把“级联”从 verifier 后处理改成 search state

现在 condition / compatibility 更像 route 生成后的检查。
未来必须让它参与搜索。

也就是说，搜索时就要判断：

```text
这个 step 加入后，当前 condition envelope 是否变窄？
是否还能和前面步骤共存？
是否需要新 stage？
是否引入 cofactor debt？
是否导致某个 enzyme/catalyst 失活风险？
```

如果某个候选反应单步分数很高，但会破坏 cascade condition envelope，它就不应该排在前面。

---

### 第二，加入 stage actions

你的 route-tree action 不能只有：

```text
choose reaction
expand leaf
```

还要有：

```text
merge current step into existing stage
start new stage
insert buffer exchange
insert isolation
insert cofactor regeneration
replace enzyme with condition-compatible enzyme
replace organic step with enzymatic bypass
replace enzymatic step with organic rescue
```

这会让 AutoPlanner 从“路线搜索器”变成“过程规划器”。

---

### 第三，建立 cascade compatibility ledger

每条 partial route 都维护一个 ledger：

```text
pH_window
temperature_window
solvent_tolerance
water_tolerance
oxygen_requirement
redox_state
cofactor_balance
metal_presence
buffer_components
enzyme_stability_risks
reactive_intermediate_risks
inhibition_flags
```

例如：

```text
NADPH consumed: 1 equivalent
NADPH regeneration: absent
aldehyde intermediate: present
amine donor: present
ADH inhibition risk: medium
DMSO fraction: 15%
enzyme solvent tolerance evidence: weak
```

这比单纯打一个 compatibility score 强得多。

---

### 第四，建立 evidence pack

实验组不会只看模型分数。
他们会问：

```text
为什么你认为这个酶可以做？
为什么这个 pH 可行？
为什么这两个步骤能 one-pot？
有没有文献、数据库、相似底物证据？
有没有 cofactor regeneration？
有没有潜在 inhibition？
```

所以每条路线必须输出：

```text
reaction evidence
enzyme evidence
condition evidence
compatibility evidence
negative evidence
uncertainty
recommended validation experiment
```

如果某个判断只是模型猜的，要明确标成：

```text
model-only, low evidence
```

这反而会增加可信度。

---

### 第五，做主动学习，而不是盲目扩大模型

数据缺失时，最先进系统一定不是“训练一次就结束”，而是：

```text
planner proposes uncertain high-value cascades
        ↓
expert reviews
        ↓
small experimental screen
        ↓
compatibility data returns
        ↓
model recalibrates
        ↓
planner improves
```

你可以把这写成：

> closed-loop cascade planning under sparse supervision

这会比单纯说“我们训练了一个 transformer”更先进。

---

## 未来 3 个月最现实的实施路线

### Month 1：定义任务和数据格式

完成：

```text
CascadeRecord schema
ConditionEnvelope schema
CompatibilityGraph schema
StagePartition schema
EvidencePack schema
```

同时人工整理第一版：

```text
100 条 high-quality cascade examples
300–1000 条 silver adjacent reaction pairs
1000 条 rule-derived incompatibility negatives
```

目标不是大，而是干净。

---

### Month 2：做 Cascade Verifier + Compatibility Model v1

先不用大模型。
先做一个强 verifier：

```text
pH interval overlap
temperature interval overlap
solvent tolerance
cofactor balance
redox conflict
metal/enzyme conflict
oxidant/reductant conflict
reactive intermediate risk
buffer conflict
one-pot vs sequential rule
```

然后训练一个轻量模型：

```text
pairwise step compatibility classifier
condition envelope ranker
stage merge/split classifier
```

这会马上让你的系统比普通 route planner 更像 cascade planner。

---

### Month 3：接入 route-tree search

把 condition state 接入 search scoring：

```text
route_score =
  retrosynthetic_score
  + step_plausibility
  + enzyme/catalyst_match
  + condition_envelope_score
  + cascade_compatibility_score
  + cofactor_closure_score
  - uncertainty_penalty
  - operation_complexity_penalty
```

然后输出两类路线：

```text
best cascade route
best non-cascade fallback route
```

这点很重要。因为真正实验系统不应该强迫所有路线 one-pot。
它应该能说：

```text
Route A 可以 one-pot 3-step cascade，但证据弱。
Route B 需要 two-stage telescoping，但成功概率更高。
Route C 不适合 cascade，只适合 stepwise synthesis。
```

这才是成熟规划。

---

## 你要避免的三个坑

### 坑 1：把“多步路线”误称为“级联”

不是所有多步路线都是 cascade。
如果每一步都需要隔离纯化，那只是 multistep synthesis，不是 cascade。

你的系统必须区分：

```text
linear multistep route
telescoped route
sequential one-pot cascade
concurrent one-pot cascade
multi-enzyme cascade
chemoenzymatic cascade
```

---

### 坑 2：只预测每一步条件，然后取交集

这太弱。

例如两个酶 pH 都兼容 7.5，不代表它们能共存。
还可能有：

```text
substrate inhibition
product inhibition
cofactor competition
oxidant/reductant conflict
metal poisoning
solvent instability
intermediate degradation
enzyme-enzyme interference
```

所以你需要 compatibility graph，而不是简单 condition overlap。

---

### 坑 3：用 pseudo-cascade 数据训练后直接宣称真实 cascade 能力

可以用 pseudo 数据，但必须分层：

```text
gold = 文献真实级联 / 实验验证
silver = pathway adjacency / database-derived likely compatible
bronze = rule-generated / model-generated pseudo cascade
negative = rule-derived or expert-labeled incompatibility
```

评估时只能用 gold / expert / wet-lab validation 做最终结论。

---

## 你和实验组专家应该这样讨论

你可以直接说：

> 我现在不想把 AutoPlanner 做成普通 CASP 系统。普通系统关注 target 到 stock 的路线；我真正想做的是 condition-constrained catalytic cascade planning。核心问题不是“能不能生成多步反应”，而是“这些催化步骤能不能在共享或分段条件下形成可执行级联”。
>
> 由于真实 cascade 数据非常稀缺，我不打算单纯训练一个端到端大模型，而是建立 CascadeDB-Gold、silver pathway data、rule-derived incompatibility negatives、condition envelope model、compatibility graph 和 active learning loop。系统会输出路线、stage partition、共享条件、cofactor/redox ledger、风险和证据，而不是只输出 reaction sequence。

这段话很重要。它会让专家明白你不是在做“AI 猜路线”，而是在做“级联催化过程规划”。

---

## 最终建议

你现在应该果断把研发路线从：

```text
better route-tree retrosynthesis
```

转向：

```text
condition-constrained cascade planning
```

普通 route-tree CASP 已经有很多强系统。你的真正机会是：

> **第一个把级联催化的 condition envelope、stage partition、compatibility graph、cofactor/redox ledger 和 uncertainty-aware active learning 放进 route-tree search 的系统。**

数据缺失不应该让你退回普通路线规划。
相反，它应该成为你的核心论文问题：

> **How to plan catalytic cascades when cascade data are scarce?**

如果你能把这个问题定义清楚，建立 CascadeBench，做出 gold/silver/negative 数据体系，并证明你的系统在 cascade feasibility ranking、condition-compatible route planning 和 expert/wet-lab hit rate 上优于普通 hybrid retrosynthesis baselines，那你就不是“接近国际前沿”，而是在定义一个新的前沿。
